## 💼 Projects

### LexGuard (2025.04 - 2025.11)
**RAG 기반 AI 계약서 법률 리스크 분석 서비스**

**Repo**: [렉스가드 깃허브](https://github.com/ggwanrok/lexguard_rag)

**시연 영상**
[![LexGuard Demo Video](https://img.youtube.com/vi/a1vg1WrbpxE/maxresdefault.jpg)](https://youtu.be/a1vg1WrbpxE)

**개발 배경**
- 계약서는 전문 용어와 복잡한 조항으로 인해 비전문가가 리스크를 파악하기 어렵고, 변호사 검토는 시간·비용 부담이 큽니다. 특히 스타트업/중소기업은 상시 법무 검토 체계를 갖추기 어려워 위험 조항을 놓치기 쉽습니다. 렉스가드는 계약 체결 전 법적 위험을 사전 식별해, 빠르고 합리적인 비용의 1차 검토를 제공하는 것을 목표로 개발했습니다. 

**주요 기능**
- 계약서 업로드 및 AI 자동 분석
- 조항별 리스크 분류 및 종합 평가 시각화
- 리스크 기반 수정 제안 생성
- 하이라이트 반영 개정 계약서 자동 생성
- 결과 PDF 다운로드 지원

**기술 스택**
- **Backend**: Python, FastAPI
- **AI / RAG**: Gemini API, Hugging Face Sentence-Transformers, RAG, Jinja2
- **Data / Search**: Qdrant
- **Async / Validation**: AsyncIO, HTTPX, Pydantic
- **Monitoring**: Python Logging

**핵심 기여**

- 단일 프롬프트 호출 방식 대신 계약서 분석 과정을 **정규화 → 벡터 검색 → 리스크 카테고리별 분석 및 병합 → 개정안 생성 → 요약** 단계로 분리한 **FastAPI 기반 RAG 오케스트레이션 구조**를 설계·구현했습니다. 이를 통해 단계별 디버깅과 유지보수가 가능한 구조를 만들었습니다.
  
- 프롬프트 템플릿과 계약 유형·관할·당사자 역할별 검토 기준을 **분석 로직과 분리해 Jinja2 + 정책 패키지 구조로 외부화**했습니다. 이를 통해 분석 로직 변경을 최소화하면서도 **정책/템플릿 단위로 검토 기준을 추가·조정할 수 있는 확장 가능한 구조**를 구축했습니다.
  
- 비정형 계약서를 그대로 검색에 활용할 경우 조항 경계가 모호해 서로 다른 내용이 섞이고, 검색 근거와 리스크 판단의 정확도가 떨어지는 문제가 있었습니다. 이를 해결하기 위해 계약서를 **조/항 단위로 정규화**해 **검색·검토·개정의 기준 단위를 일치**시켰고, 그 결과 조항-근거 매칭 품질과 분석 결과의 명확성을 높였습니다.

- 초기의 **범용 임베딩 기반 의미검색** 방식은 계약 유형, 관할, 핵심 키워드를 충분히 반영하지 못해 검색 품질에 한계가 있었습니다. 이를 개선하기 위해 **Hugging Face 기반 법률 특화 임베딩 모델을 비교·선정**하고, **Qdrant 벡터 검색과 Payload Filter(계약유형/관할/키워드)** 를 결합해 맥락 분산을 완화하고 조항-근거 매칭 품질을 개선했습니다.

- 분석 및 개선안 생성 단계에서 why / trigger / citation 기반 **근거 제시와 추천안-리스크 연관성 검증을 도입**해, 일반론적이거나 근거가 부족한 응답을 줄이고 **설명 가능한 리스크 분석 체계**를 구축했습니다.

- 조항별 분석과 조항 내 리스크 관점별 분석을 **이중 비동기 병렬 처리** 구조로 구성해, 선형 분석 대비 **57%(7분 → 3분)** 단축했습니다. 또한 병렬 실행 이후 **인덱스 기준 재정렬**을 적용해 응답 순서의 정합성을 보장했습니다.

- 외부 LLM 호출은 **Gemini SDK를 기본 경로, REST 호출을 폴백 경로**로 구성해 장애 상황에서도 분석이 중단되지 않도록 했습니다. 또한 **UUID 기반 오류 추적, Header Lock, 수정 불필요 조항에 대한 개정 억제 로직**을 적용해 운영 안정성과 문서 일관성을 확보했습니다.

### 숭민청원 (2025.03 - 2025.06)
**학생 의견을 청원과 행정 대응 흐름으로 연결한 자동화 서비스**

**Repo**: [숭민청원 깃허브](https://github.com/SoongMinChungWon/SoongMin_spring_server)

**시연영상**
[![숭민청원 Demo Video](https://img.youtube.com/vi/lQNtFkVswys/maxresdefault.jpg)](https://youtu.be/lQNtFkVswys)

**개발 배경**
- 기존 개인 민원이나 비정형 커뮤니티 방식은 공감 수가 체계적으로 집계되지 않고, 동일 이슈가 여러 글로 분산되며, 감정적 논쟁·익명성으로 인한 소음과 책임 회피로 실질적 개선으로 이어지기 어렵다는 한계를 가져왔습니다. 이에 신뢰성과 효율성을 갖춘 플랫폼이 필요하다고 판단하여, 학생들의 생활 속 불편·개선 요구를 집단적으로 모으고, 공감이 확인된 사안만을 학교에 공식 전달하는 민주적 의사소통 창구를 구축하고자 했습니다.

**주요 기능**
- 재학생 인증 기반 로그인 및 자동 회원가입
- AI 청원문 작성 보조
- 참여 기준 충족 시 청원 상태 전환 및 담당 부서 메일 자동 발송
- 회신 메일 자동 수집 및 답변 상태 반영
- 임베딩 기반 유사 청원 검색
- 교내 뉴스 카드형 제공

**기술 스택**
- **Backend**: Java, Spring Boot
- **AI**: OpenAI API
- **Mail**: Spring Mail (SMTP), Jakarta Mail (IMAPS)
- **Data / Search**: MySQL, Qdrant
- **Infra / DevOps**: Docker, Docker Compose, GitHub Actions, AWS EC2

**핵심 기여**

- **Spring Boot 백엔드 구조와 ERD를 설계**해, 청원 등록부터 참여, 메일 발송, 답변 반영까지의 서비스 흐름을 안정적으로 담을 수 있는 기반을 구축했습니다.

- 학생들이 보다 정돈된 형식으로 청원문을 작성할 수 있도록 **OpenAI API 기반 자동완성·문장 다듬기 기능**을 연동했습니다. 이를 통해 자유 서술형 의견을 보다 공식적인 청원문 형태로 보조해, 단순 불만 게시글이 아니라 실제 전달 가능한 안건으로 정리될 수 있게 했습니다.

- 단순 게시 후 방치되는 구조를 넘어서기 위해, **참여자 수·동의율 조건 충족 시 게시글 상태를 자동 전환하고, SMTP 기반의 담당 부서 메일을 자동 발송하는 처리 흐름**을 구현했습니다. 이를 통해 학생 의견이 실제 행정 대응으로 연결되는 자동화된 파이프라인을 만들었습니다.

- 담당 부서의 회신이 서비스에 수동 반영되는 비효율을 줄이기 위해, **Jakarta Mail(IMAPS) 기반 메일 모니터링**으로 회신을 주기적으로 수집·파싱하고, 답변 저장 및 게시글 상태 업데이트까지 자동화했습니다. 이를 통해 청원 접수 이후 처리 현황을 사용자에게 지속적으로 보여줄 수 있도록 했습니다.

- 기존 키워드 검색으로는 유사 이슈가 분산되는 문제가 있어, **게시글 등록 시 임베딩 생성 및 벡터 저장, 검색 시 쿼리 임베딩 기반 유사도 조회**로 이어지는 의미 기반 검색 파이프라인을 구현했습니다. 이를 통해 단순 단어 일치가 아닌 맥락 중심으로 유사 청원을 탐색할 수 있도록 개선했습니다.

- **Docker Compose**와 **GitHub Actions**를 활용해 **CI/CD 파이프라인**을 구축하였으며, **AWS EC2 환경**에 안정적으로 배포할 수 있도록 전체 인프라를 설계하고 운영하였습니다.
